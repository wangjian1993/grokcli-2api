## 分析

请求连接数太多 默认vite8策略下 请求静态资源需要达到100+请求 而每次的连接数是有限的

极小的碎片文件(0.x 1.x KB)文件 大量占用了连接数

通过减少连接数的办法来提升加载速度 这样虽然会造成体积的增加 因为很多包打在一个文件了速度反而比之前更快

但是注意: **nginx必须开始gzip** 这样体积经过压缩能和不分块差不多(实测多了500K)

## 策略

`internal/vite-config/src/config/code-split.ts`

- 允许较大的体积生成 但是要关注瀑布流情况 如果有大的js阻塞了后续请求 需要进一步拆分
- 尽量减少首屏的碎片文件生成

## 注意

业务模块的增加可能也会影响首屏体积 可能需要定期检查
